英语俱乐部交流平台

申长硕PB22020518 & 滕润喆PB22061212

引言

开发意义

随着信息技术的不断进步，越来越多的学校开始建设信息化平台。 作为英语俱乐部的往届的骨干，结合课程特性创建一个英语俱乐部交流平台尤为重要。
由于参与者和活动数量庞大，手工管理俱乐部的交流和活动变得越来越不切实际。 利用计算机技术进行现代信息管理可以显著提高这些操作的准确性和效率。

可行性

传统的俱乐部交流和活动管理常常存在准确性低、效率低等问题。
计算机具有存储快、检索方便、准确性高的特点，可以有效解决这些问题，使管理过程更加顺畅和可靠。

应用环境
  • 架构： B\S (Browser/Server) 架构
  • 数据库： mysql Ver 14.14 Distrib 5.7.24, for Linux (x86_64) using EditLine wrapper
  • 技术路线： 使用 Django 框架
  • 开发工具： 使用 PyCharm 作为主要的开发环境，结合 Git 进行版本控制和协作开发
  • 辅助技术： 使用爬虫技术，获取初始的寻宝体验的谜题
获取源码

所有程序已托管到 `github` 平台，可以直接利用如下命令获取：

git clone git@github.com:Stephen-Shenc/USTC-English-Club.git

请确保配置相关数据库后才可执行。

补充

范式分析

  • 1NF：所有属性值都是原子的，不可再分，所以我们的模型设计符合1NF

  • 2NF：由于User、Post、Comments、Reply、Like、Riddle、Prize均有主键，且所有非主属性完全依赖于主键，因此该也符合第二范式

  • 3NF：我们设计的模型中各实体的非主属性都直接依赖于主键，没有传递依赖，所以符合第三范式

  • 4NF——是否有必要升4NF的考量：

    多值依赖是在一个关系中，如果一个属性集X确定另一个属性集Y，并且Y是X的独立多值属性，那么我们说Y多值依赖于X，记作X →→ Y。多值依赖的一个关键点是，一个属性集可以独立地与多个值关联，而不依赖于关系中的其他属性。

    举个例子，假设我们有如下的表结构，其中一个用户可能会对多个帖子点赞和评论：

    原始表的情况

    User表

    user_id name likes_num comments_num
    1 Runge 2 2
    2 Shenc 3 1

    在这个表中，likes_num和comments_num分别表示用户点赞和评论的总数。这些数据是从Likes表和Comments表聚合得到的。

    Likes表
    user_id post_id
    1 101
    1 102
    2 103
    2 104
    2 105
    Comments表
    user_id post_id
    1 101
    1 103
    2 106

    在原始User表中，每个user_id唯一确定了likes_num和comments_num，但这些数据是通过聚合Likes和Comments表中的数据得来的。likes_num是通过对Likes表中对应user_id的记录进行计数得到的。comments_num是通过对Comments表中对应user_id的记录进行计数得到的。可以理解如下：

    user_id name likes comments
    1 Runge 101, 102 101, 103
    2 Shenc 103, 104, 105 106

    即存在多值依赖：user_id →→ likes和user_id →→ comments，那原关系也就不满足第四范式。

    若改进，我们想到的设计如下：

    User表

    CREATE TABLE User (
                                user_id INT PRIMARY KEY,
                                name VARCHAR(255)
                            );

    Likes表

    CREATE TABLE Likes (
                                user_id INT,
                                post_id INT,
                                PRIMARY KEY (user_id, post_id),
                                FOREIGN KEY (user_id) REFERENCES User(user_id)
                            );

    Comments表

    CREATE TABLE Comments (
                                user_id INT,
                                post_id INT,
                                PRIMARY KEY (user_id, post_id),
                                FOREIGN KEY (user_id) REFERENCES User(user_id)
                            );

    通过这种方式，我们消除了多值依赖，因为User表只存储用户基本信息，Likes表独立记录用户点赞的帖子，Comments表独立记录用户评论的帖子。然而，这种设计有一个显著的缺点，就是每次需要展示点赞数和评论数时，都需要重新查询和计算，如果用户数量和互动频繁的情况下，这种效率就比较低了。因此，我们认为在实际应用中，仍然保留在第三范式更为合理。通过这种trade-off，我们觉得能够更好地满足用户需求，提供优质的系统体验。

安全性考量

  • 身份验证：在用户注册的时候和忘记密码进行修改密码的时候均使用邮箱发送动态验证码进行身份验证。

  • 敏感数据加密：对于用户的密码，我们使用Django提供的make_password和check_password进行加密存储及验证，数据库中存储的密码均值哈希加密之后的密码。

  • 权限控制：我们自创的身份验证（当然Django已经内置了对应的功能），确保只有登陆的用户参能参与论坛的点赞，发帖，评论，回复等操作。以及确保只有管理员有权限进入Admin-Panel进行数据总揽操纵。

  • CSRF保护：我们在所有表单提交的部分均使用了跨站点请求(SCRF)保护，从而确保用户提交的表单数据是合法的。

  • 日志记录：我们通过设置日志记录(logging)系统以记录重要的操作和事件，例如在项目推进过程中的错误和异常、请求和响应、调试信息、性能监视、安全事件及系统和环境信息等。这样便于追踪和排查潜在的安全问题。

    （性能方面，助教老师在线下汇报的时候还问我们有一个地方加载的比较慢，那一次是因为用腾讯会议共享屏幕的原因，从目前的测试情况看，我们已经做出来的各功能的完成性能是不影响用户体验的）。

ps：在和一位网安学长的交流中和数据库课程上，我们体会到了平台对于自身数据库保护以及对于用户信息保护的重要性和措施，（例如密码加密方面，我们很早的时候在学长那里了解到了MD5之类的一些加密方法，之后选用Django框架之后发现已经有加密的方法就直接拿来用了）所以在项目推进的过程中始终谨记这一点！

事务并发考量

在事务并发可能导致的冲突方面，我们主要使用Django本身提供的事务管理工具进行解决，主要使用两种方法：

  • atomic保证某代码块的原子性，确保代码块在一个事物内执行
  • select_for_update保证选中的行直到事务结束

在脏读处理方面，由于我们保证每一个基本事务的原子性，每一个基本事务都会在提交完成才结束，这个时候才有其他事务(函数)执行，所以是可以有效防止脏读的。

在另两种问题上，我们并没有做更多限制，主要原因是我们觉得在用户A浏览一个页面的时候，用户B进行评论(UPDATE抑或是INSERT)之后A不能马上看到，这是一件可以接受的事情。在下次刷新的时候可以看见就够用了，一般不会影响用户的使用体验。

故而，我们最终的隔离级别停留在Read Committed级别，通过保证特定代码块/函数的原子性，保证一个事务必须等另一个事务提交之后才能读取数据。
